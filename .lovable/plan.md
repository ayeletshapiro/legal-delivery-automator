## מטרה
כשה-AI לא מזהה לקוח בהודעה נכנסת, המערכת תשלח לאביעד הודעת WhatsApp חוזרת עם 3 אפשרויות, ותפעל לפי תשובתו.

## זרימת המשתמש

1. אביעד שולח הודעה ללא שם לקוח ברור: *"שליחות לבית משפט מחר 50 שח"*
2. המערכת יוצרת `delivery` במצב המתנה (לא נכתב לאף גיליון) ושולחת לו WhatsApp:

   ```
   לא זיהיתי לקוח עבור ההודעה:
   "שליחות לבית משפט מחר 50 שח"

   איך לשייך? ענה/י באחת הדרכים:
   • שם הלקוח (לדוגמה: כהן ושות')
   • "מזדמנים" – לשמור תחת לקוחות מזדמנים
   • "חדש: <שם>" – ליצור לקוח חדש בשם הזה
   ```

3. אביעד עונה. המערכת מזהה שזו תשובה לבירור פתוח ומבצעת:
   - **שם קיים / כינוי קיים** → משייכת ל-delivery, כותבת לגיליון של הלקוח, ושומרת את הטקסט המקורי בעצם הבעיה (לא נוסיף alias אוטומטית — כדי לא לזהם, נעשה את זה רק אם הוא ביקש "חדש").
   - **"מזדמנים"** → משייכת ללקוח מזדמנים וכותבת לגיליון שלו.
   - **"חדש: כהן"** → יוצרת לקוח חדש "כהן", מייצרת לו גיליון אוטומטית (`createSheetForClient` קיים), משייכת וכותבת.
   - **טקסט שלא מתאים לאף תבנית** → ננסה לפרש כשם לקוח חופשי; אם לא נמצא לקוח/כינוי תואם, שולחים שוב את אותה הודעת בירור.

4. אישור חוזר אחרי טיפול:
   *"נשמר עבור 'כהן ושות'' ✓"* או *"נוצר לקוח חדש 'כהן' ונשמר ✓"*.

## טיים-אאוט
- בירור פתוח נשאר פעיל **24 שעות**.
- אחרי 24 שעות (בכל עיבוד הודעה חדשה, בודקים lazily): המערכת סוגרת את הבירור, משייכת אוטומטית את ה-delivery ל"מזדמנים" וכותבת לגיליון. נרשמת שגיאה ב-`processing_errors` ("בירור פג תוקף, שובץ למזדמנים").

## שינויים טכניים

### 1. טבלה חדשה `pending_clarifications`
- שדות: `user_id`, `message_id` (FK ל-`incoming_messages`), `delivery_id` (FK ל-`deliveries`), `raw_text`, `created_at`, `resolved_at`, `resolution` (text nullable: 'matched'|'misc'|'created'|'expired').
- אינדקס חלקי: `(user_id) WHERE resolved_at IS NULL`.
- RLS: user רואה את שלו; service_role מלא; GRANTs ל-authenticated + service_role.

### 2. עדכון `processing.server.ts` במסלול `missing_client`
- ה-delivery עדיין נוצר עם `client_id = מזדמנים` כפלייסהולדר, אבל `write_status = 'awaiting_clarification'` ו**לא** נכתב לגיליון.
- נפתחת רשומה ב-`pending_clarifications`.
- נשלחת הודעת WhatsApp לאביעד (ראה §4).
- אם שליחת WhatsApp נכשלה — fallback: כותבים בכל זאת לגיליון מזדמנים ורושמים שגיאה.

### 3. עדכון `twilio-webhook.ts`
לפני קריאה ל-`processIncomingMessage`:
- אם למשתמש יש `pending_clarifications` פתוח שלא פג (≤24h): מטפלים בהודעה הזו כתשובת בירור — מנתחים את הטקסט (שם / "מזדמנים" / "חדש: X"), מבצעים את הפעולה על ה-delivery הממתין, מסמנים `resolved_at`+`resolution`, ושולחים הודעת אישור. **לא** יוצרים `delivery` חדש.
- אם הבירור הפתוח פג תוקף (>24h) — סוגרים אותו ('expired'), משייכים את ה-delivery הקודם למזדמנים + כותבים, וממשיכים לעבד את ההודעה החדשה רגיל.
- אם אין בירור פתוח — עיבוד רגיל.

### 4. שליחת WhatsApp יוצא — `src/lib/twilio.server.ts`
פונקציה `sendWhatsAppMessage(toPhoneE164, body)` שמשתמשת ב-Twilio API (אותו מספר נכנס/יוצא, מוגדר ב-`app_config`/secrets). מספר היעד נלקח מ-`profiles.whatsapp_phone` של בעל ה-message.

### 5. עדכון UI (קטן)
- במסך "הודעות": סטטוס חדש "ממתין להבהרה" עם תווית.
- במסך "משלוחים": אותו דבר ב-`write_status`.

## תרחישי קצה
- שני בירורים פתוחים זה אחרי זה → התשובה מתייחסת ל**אחרון** הפתוח (לפי `created_at DESC`). הקודמים נסגרים אוטומטית כשהם פגים.
- "חדש: כהן" אבל "כהן" כבר קיים פעיל → לא ניצור כפילות; נשייך לקיים ונודיע: *"נמצא לקוח קיים 'כהן' — שובץ אליו"*.
- שליחת WhatsApp נכשלת → fallback למזדמנים, נרשמת שגיאה ב-`processing_errors`.
- אביעד לוחץ "עבד מחדש" מה-UI בזמן שיש בירור פתוח → הריצה החוזרת לא תשנה כלום אם הסטטוס הוא `awaiting_clarification` (היא תעצור בפתח). אם בינתיים הוסיף את הלקוח ידנית — הריצה החוזרת תזהה ותסגור את הבירור אוטומטית.
